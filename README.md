# Warehaus :package:

Warehaus is a web based inventory management prototype that helps warehouse admins and staff track stock, log borrowings and returns, and produce real time inventory reports. The system exposes both a full Blade based web UI and a versioned REST API.

## Features

### Core

- **Authentication and Authorization**. Laravel Breeze scaffold combined with role based access control (Admin, Staff, Manager) enforced by `RoleMiddleware` and policies.
- **Product Management**. Full CRUD for products including code, category, stock, location, condition (`baik`, `rusak_ringan`, `rusak_berat`), and image upload.
- **Category Management**. Admin only CRUD for categories.
- **Borrowing System**. Multi item borrowing records with stock validation wrapped in a database transaction and row level locking so stock cannot go negative under concurrent writes.
- **Return Processing**. Stock is restored automatically when items are marked as returned.
- **Dashboard**. Summary cards (total stock, currently borrowed, available, category count), a 12 month borrowing trend chart (Chart.js), latest borrowings table, low stock table, and an overdue list.
- **User Management**. Admin only CRUD for users and role assignment.

### Additional

- :page_facing_up: **PDF export** of inventory reports and borrowing history via `barryvdh/laravel-dompdf`.
- :bar_chart: **Excel export** with two sheets (Inventory, Borrowings) via `maatwebsite/excel`.
- :new_moon_with_face: **Dark mode toggle** driven by Alpine.js, persisted in `localStorage`, with a `prefers-color-scheme` fallback.
- :bell: **Low stock notification banner** on the dashboard when any product falls to five units or fewer.
- :electric_plug: **REST API v1** authenticated by Laravel Sanctum, using Eloquent API Resources and a consistent `{ success, message, data }` envelope.
- :book: **OpenAPI documentation** generated by `l5-swagger`, browsable through Swagger UI.
- :ship: **Deployment ready** with `Procfile`, `nixpacks.toml`, and a production Dockerfile.

## Database Schema

The database schema is defined in DBML in [`schema.dbml`](./schema.dbml). Paste that file into [dbdiagram.io](https://dbdiagram.io/) to render an interactive ERD.

Table overview:

| Table               | Purpose                                                     |
|---------------------|-------------------------------------------------------------|
| `roles`             | Role master data (admin, staff, manager)                    |
| `users`             | Application accounts with a `role_id` foreign key           |
| `categories`        | Product categories                                          |
| `products`          | Product records (code, stock, condition, location, image)   |
| `borrowings`        | Borrowing header (borrower, dates, status, notes)           |
| `borrowing_details` | Line items per borrowing (product and quantity)             |

## Local Installation

### Prerequisites

- PHP 8.3 or newer (8.5 recommended)
- Composer 2.x
- Node.js 20 or newer with npm
- A PostgreSQL instance (local or Supabase)

### Setup

```bash
# 1. Clone the repository
git clone https://github.com/<username>/inventory-app.git
cd inventory-app

# 2. Install PHP dependencies
composer install

# 3. Install frontend dependencies
npm install

# 4. Copy the environment file
cp .env.example .env

# 5. Generate the application key
php artisan key:generate

# 6. Configure the database section of .env
# DB_CONNECTION=pgsql
# DB_HOST=<postgres-host>
# DB_PORT=6543
# DB_DATABASE=postgres
# DB_USERNAME=<user>
# DB_PASSWORD=<password>

# 7. Run migrations with seed data (roles, users, categories, products)
php artisan migrate:fresh --seed

# 8. Create the storage symlink (required for product image uploads)
php artisan storage:link
```

### Running the App

```bash
# Terminal 1: Laravel dev server
php artisan serve

# Terminal 2: Vite HMR
npm run dev
```

Open `http://localhost:8000`.

## Docker Compose (Optional)

The Docker Compose stack runs three services: `app` (FrankenPHP built from [Dockerfile](./Dockerfile)), `nginx` (reverse proxy on port 8080), and `db` (PostgreSQL 16). Nginx configuration lives in [`docker/nginx/default.conf`](./docker/nginx/default.conf).

### From Scratch to Serving

```bash
# 1. Copy the environment file (Docker overrides DB_HOST and DB_PORT via compose)
cp .env.example .env

# 2. Build images and start the stack
docker compose up -d --build

# 3. Generate APP_KEY inside the container
docker compose exec app php artisan key:generate

# 4. Migrate and seed
docker compose exec app php artisan migrate --seed --force

# 5. Symlink storage for product images
docker compose exec app php artisan storage:link
```

Open `http://localhost:8080`. See the [Test Accounts](#test-accounts) section for credentials.

### Everyday Commands

```bash
docker compose logs -f app                          # tail Laravel logs
docker compose exec app php artisan test --compact  # run the test suite inside the container
docker compose exec app php artisan tinker          # Laravel REPL
docker compose exec db psql -U warehaus warehaus    # open psql directly
docker compose down                                 # stop the stack, keep the DB volume
docker compose down -v                              # stop the stack and drop the volume (fresh start)
```

PostgreSQL data lives in the named volume `warehaus_pgdata`, so it survives container restarts.

## REST API

Base URL: `/api/v1`. Every endpoint is protected by a Sanctum bearer token except `POST /login`.

### Interactive Documentation

Swagger UI is served at `http://localhost:8000/api/documentation` and reads `storage/api-docs/api-docs.json`. Regenerate it after editing annotations:

```bash
php artisan l5-swagger:generate
```

To regenerate the spec on every request during development, set `L5_SWAGGER_GENERATE_ALWAYS=true` in `.env`.

### Issue a Token

```bash
curl -X POST http://localhost:8000/api/v1/login \
  -H "Accept: application/json" \
  -H "Content-Type: application/json" \
  -d '{"email":"admin@warehaus.test","password":"password","device_name":"postman"}'
```

Response:

```json
{
  "success": true,
  "message": "Login successful.",
  "data": {
    "token": "1|xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx",
    "user": { "id": 1, "name": "Admin", "email": "admin@warehaus.test", "role": "admin" }
  }
}
```

Include the token on subsequent requests: `Authorization: Bearer <token>`.

### Endpoints

| Method | URL                                | Description                             |
|--------|------------------------------------|-----------------------------------------|
| POST   | `/api/v1/login`                    | Issue a Sanctum token                   |
| POST   | `/api/v1/logout`                   | Revoke the active token                 |
| GET    | `/api/v1/user`                     | Return the authenticated user           |
| GET    | `/api/v1/products`                 | List products (paginated with filters)  |
| POST   | `/api/v1/products`                 | Create a product                        |
| GET    | `/api/v1/products/{id}`            | Product detail                          |
| PUT    | `/api/v1/products/{id}`            | Update a product                        |
| DELETE | `/api/v1/products/{id}`            | Delete a product                        |
| GET    | `/api/v1/borrowings`               | List borrowings                         |
| POST   | `/api/v1/borrowings`               | Create a multi item borrowing           |
| GET    | `/api/v1/borrowings/{id}`          | Borrowing detail                        |
| PUT    | `/api/v1/borrowings/{id}`          | Update a borrowing (notes, due date)    |
| DELETE | `/api/v1/borrowings/{id}`          | Delete a completed borrowing            |
| PATCH  | `/api/v1/borrowings/{id}/return`   | Process a return                        |
| GET    | `/api/v1/categories`               | List categories                         |
| GET    | `/api/v1/categories/{id}`          | Category detail                         |
| GET    | `/api/v1/dashboard/stats`          | Aggregated dashboard statistics         |

Every response follows the same envelope:

```json
{
  "success": true,
  "message": "...",
  "data": { }
}
```

## Testing

Tests run against an in memory SQLite database (configured in `phpunit.xml`), so they never touch your production data.

```bash
# Run the full test suite
php artisan test --compact

# Run a single file
php artisan test --compact tests/Feature/ExampleTest.php

# Filter by test name
php artisan test --compact --filter=test_name
```

> The `pdo_sqlite` PHP extension must be enabled. On Windows, uncomment `extension=pdo_sqlite` in your active `php.ini`.

Format PHP files with Pint before committing:

```bash
vendor/bin/pint --dirty --format agent
```

## Deployment (Railway)

1. Push the repository to GitHub.
2. Create a new project on [Railway](https://railway.app) and choose **Deploy from GitHub repo**.
3. Set the following environment variables in the Railway dashboard:
   ```
   APP_KEY=<output of: php artisan key:generate --show>
   APP_URL=https://<railway-domain>
   DB_CONNECTION=pgsql
   DB_HOST=<postgres-host>
   DB_PORT=6543
   DB_DATABASE=postgres
   DB_USERNAME=<user>
   DB_PASSWORD=<password>
   ```
4. Railway will pick up `nixpacks.toml` for the build and `Procfile` for the start command.
5. After the first deploy, run `php artisan migrate --force` from the Railway shell.
